# 업무운영 Start

이 문서는 프로젝트 `DOCS/운영업무`의 부트스트랩 문서입니다. 앞으로 어떤 AI가 들어오더라도 최소한 이 파일 하나만 먼저 읽으면, 운영업무 체계의 목적과 역할, 각 하위 문서의 의미, 문서 간 이동 규칙, 사용자가 기대하는 자율 운영 방식까지 이해할 수 있어야 합니다.

## 이 문서만으로 알아야 할 핵심

- 사용자는 AI가 장시간 자율적으로 일하면서 결과물을 계속 생산하길 원한다.
- 단순 코딩이 아니라 앱 개발, 문서화, 운영업무, 나중에는 사업 운영까지 이어질 수 있다.
- AI는 비전을 읽고 현재 결과물과 비교해 진척을 판단한 뒤 아젠다, Todo, Task를 만들어야 한다.
- 이 문서는 전체 구조의 입구이며, 각 하위 문서로 들어가기 전 방향을 맞추는 기준점이다.

## 업무계획이란 무엇인가

업무계획은 AI가 실제로 앞으로 밀고 나갈 실행 축입니다. 쉽게 말해 `무엇을 목표로 보고`, `지금 무엇을 논의하고`, `무엇을 하기로 결정했고`, `지금 당장 무엇을 실행할지`를 시간 흐름에 따라 정리하는 문서 묶음입니다.

- 업무계획은 방향에서 실행으로 내려가는 축이다.
- 비전 -> 아젠다 -> Todo -> Task 순서로 점점 구체화된다.
- 아래로 갈수록 AI 자율성과 생산성이 커진다.
- 위 문서와 충돌하는 실행은 아래 문서에서 멈춰야 한다.

## 업무맥락이란 무엇인가

업무맥락은 AI가 업무계획을 세우기 전에 참고하는 판단 재료입니다. 쉽게 말해 `지금 실제로 어디까지 왔는지`, `무슨 후보들이 남아 있는지`, `어떤 증거와 공백이 있는지`를 보는 문서 묶음입니다.

- 업무맥락은 실행 명령서가 아니라 판단 근거다.
- 현황은 증거판이고, 백로그는 후보 저장소다.
- 맥락 없이 계획만 세우면 Todo와 Task가 비전에서 이탈할 수 있다.
- 업무맥락은 기획 루프의 품질을 잡아주는 보정 장치다.

## 최상위 목적

- 사용자의 비전과 현재 결과물을 비교해 부족한 지점을 찾는다.
- 그 차이를 메우기 위한 아젠다, Todo, Task를 계속 갱신한다.
- Task 단위까지 내려가면 실제 코딩, 문서 작성, 운영 작업을 생산한다.
- 생산 이후에는 다시 현황을 갱신해 다음 루프의 판단 재료로 남긴다.

## 디렉터리 규칙

- 운영업무 관련 문서는 모두 `DOCS/운영업무/` 아래에 둔다.
- 시작 문서는 항상 `DOCS/운영업무/업무운영_start.md` 이다.
- 실행 문서는 `DOCS/운영업무/업무계획/` 아래에 둔다.
- 판단 문서는 `DOCS/운영업무/업무맥락/` 아래에 둔다.
- 새 AI는 다른 문서보다 먼저 `업무운영_start.md`를 읽는다.

## 문서 체계

- 업무운영_start.md
- 업무계획/
  - 업무-비전.md
  - 업무-아젠다.md
  - 업무-Todo.md
  - 업무-Task.md
- 업무맥락/
  - 업무-백로그.md
  - 업무-현황.md

## 각 문서의 역할

### 업무운영_start.md
- 전체 운영업무 구조의 시작점이다.
- 새 AI가 가장 먼저 읽는 네비게이션 문서다.
- 각 하위 폴더와 문서의 역할, 읽는 순서, 운영 원칙을 설명한다.

### 업무계획/업무-비전.md
- 사용자의 목표, 금지사항, 완료 기준을 정의한다.
- 왜 이 프로젝트를 하는지와 무엇을 절대 잃지 말아야 하는지 담는다.
- 새 AI는 여기서 방향성과 금지선을 먼저 파악해야 한다.

### 업무계획/업무-아젠다.md
- 지금 이 시점에서 가장 중요한 논점과 결정 필요 항목을 정리한다.
- 아직 실행으로 확정되지 않은 문제를 다루는 공간이다.
- Todo로 넘기기 전, 여기서 우선순위와 판단 기준을 정리한다.

### 업무계획/업무-Todo.md
- 하기로 확정한 작업 묶음을 관리한다.
- 후보가 아니라 실제 추진할 일만 올라온다.
- 각 Todo는 완료 기준과 의존성, 우선순위를 가져야 한다.

### 업무계획/업무-Task.md
- Todo를 바로 실행 가능한 최소 단위로 쪼갠다.
- 실제 코딩, 문서 작성, 검증, 운영 작업이 이루어지는 실행층이다.
- 기대 산출물과 검증 방법이 없는 Task는 미완성 Task로 본다.

### 업무맥락/업무-백로그.md
- 아직 확정하지 않은 후보, 아이디어, 개선안을 담는다.
- Todo를 오염시키지 않으면서 가치 있는 후보를 보관하는 완충층이다.
- 나중에 아젠다나 Todo로 승격할 수 있는 재료를 모은다.

### 업무맥락/업무-현황.md
- 현재 결과물과 진척도를 증거 기반으로 기록한다.
- 무엇이 끝났고 무엇이 비었는지, 어떤 테스트와 운영 근거가 있는지 남긴다.
- 비전 대비 현재 위치를 판단하는 핵심 근거 문서다.

## 문서 간 흐름

1. `업무운영_start.md`를 읽고 전체 구조와 규칙을 파악한다.
2. `업무계획/업무-비전.md`를 읽고 사용자의 방향과 금지사항을 이해한다.
3. `업무맥락/업무-현황.md`를 읽고 실제 결과물, 빈 구간, 진척도를 파악한다.
4. 필요하면 `업무맥락/업무-백로그.md`를 읽어 아직 확정되지 않은 후보를 확인한다.
5. `업무계획/업무-아젠다.md`에 지금 중요한 논점을 정리한다.
6. `업무계획/업무-Todo.md`에 하기로 확정한 작업 묶음을 올린다.
7. `업무계획/업무-Task.md`에서 바로 실행 가능한 최소 단위로 쪼갠다.
8. 생산 후에는 다시 `업무맥락/업무-현황.md`에 증거를 남겨 다음 루프에 반영한다.

## 미래 AI를 위한 읽기 규칙

- 폴더에 문서가 있다고 자동으로 다 읽었다고 가정하지 말고, 필요한 문서를 명시적으로 연다.
- 이 파일은 요약이자 네비게이션이다. 세부 내용은 반드시 해당 하위 문서에서 다시 확인한다.
- 비전 없이 Task만 읽고 바로 실행하지 않는다.
- 현재 상태 판단 없이 Todo를 무분별하게 늘리지 않는다.
- 완료 후에는 결과와 근거를 현황에 남긴다.

## 관점 태그 메모

- 기반/관리/성장/현장은 고정 칸이라기보다 관점 태그에 가깝다.
- 하나의 일은 기반+성장, 기반+현장처럼 복합 관점을 가질 수 있다.
- 같은 일도 비전에서는 성장 중심이었다가 아젠다에서는 현장 중심으로 이동할 수 있다.
- 필요하면 주관점 1개와 보조관점 여러 개를 함께 기록한다.

## 운영 원칙

- 비전은 사용자가 강하게 잡는다.
- 아래 단계로 갈수록 AI 자율성이 커진다.
- 백로그는 Todo를 오염시키지 않으면서 아이디어를 보관하는 완충층이다.
- 현황은 비전 대비 진척을 판단하는 증거 문서다.
- 시작 문서는 항상 루트에 두고, 계획과 맥락은 각 폴더에 나눈다.
- 이 문서만 읽어도 구조와 의도를 파악할 수 있어야 한다.

## 추후 확장 메모

- 업무 문서 안에는 기반/관리/성장/현장 같은 관점이 더 세부적으로 들어갈 수 있다.
- 이후 기획, 참조자료, 참조 링크, 학습허브, 학습네트워크 같은 하위 주제도 확장 가능하다.
- 다만 처음에는 문서를 너무 잘게 쪼개지 말고, 큰 흐름을 유지한 채 필요한 것만 분리한다.
